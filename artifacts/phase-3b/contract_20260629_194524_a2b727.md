## O0 resolution (conviction_min default) — RESOLVED

The brief's open question O0 asks whether conviction is enforced only in the
prompt, and whether default-40 is byte-preserving. Findings:

1. Conviction is enforced ONLY in `ICT_SYSTEM_PROMPT` (reasoning_agent.py).
   `validate_rr` (alert.py) has NO conviction check — it only recomputes R:R
   from geometry and downgrades sub-1:1 / degenerate setups.
2. Audited `workspace/trade_traces.jsonl`: all 6 taken trades (bias != no_trade)
   have conviction ∈ {62, 68, 72}. Zero taken trades have conviction < 40.
   (8 alerts have conviction > 0; 2 of those are no_trade alerts the LLM
   emitted with nonzero conviction — irrelevant, since validate_rr
   short-circuits on no_trade before any conviction check.)
3. Audited every test fixture: all taken-trade alerts in test_alert.py,
   test_trading_loop.py, test_golden_integration.py, test_engine.py,
   test_outcome.py use conviction ≥ 60.

**Conclusion**: A Python conviction gate with default 40 is byte-identical to
today's behavior — it is a no-op for every current input (the prompt already
guarantees conviction ≥ 40 on taken trades). Default `conviction_min = 40`.
The gate activates only with a non-default conviction_min (Tier-2 recording)
or when the LLM violates the prompt. This is the O0-prescribed outcome.

## Key design decisions (documented for review)

1. **rescore filter scope**: Filters (conviction_min, rr_min, allowed_models,
   allowed_killzones) apply ONLY to traces with `outcome.result in ("win",
   "loss")` — realized trades. Traces already recorded as no_trade / no_fill /
   expired / cancelled are untouched (they have actual_rr=None → 0 in
   expectancy regardless; downgrading them changes neither expectancy nor the
   win/loss taken_trades count). This satisfies the blueprint's "no_trade /
   no_fill / expired stays whatever it is" criterion and the test strategy's
   "pre-existing no_trade/no_fill traces untouched."

2. **taken_trades** (for the min_trades guard) = count of re-scored traces with
   `result in ("win", "loss")` — trades with a realized R. This is the
   meaningful sample for estimating expectancy.

3. **expectancy** = PerformanceReport._expectancy (mean actual_rr over ALL
   traces, None→0). Single source of truth: objective REUSES
   PerformanceReport's computation, does not recompute the mean independently.

4. **allowed_models / allowed_killzones**: `None` = "allow all" (baseline);
   empty frozenset = "allow nothing" (testable). Avoids empty-set ambiguity.

5. **rr recompute** in rescore uses the SAME formula as validate_rr
   (entry_mid = (ez[0]+ez[1])/2; risk = |entry_mid - stop|;
   rr = |target - entry_mid| / risk) — imported/shared, not duplicated.

6. **Cooldown is NOT a PostLLMParams field** (documented limitation): re-scoring
   treats alerts as independent and ignores cooldown re-interaction. Cooldown
   is a TradingConfig (recording-time, Tier-2) knob only.

7. **Walk-forward**: rolling folds by timestamp, half-open intervals
   [start, start+train_days) / [start+train_days, start+train_days+test_days).
   Train∩test = ∅ per fold (asserted). Baseline OOS uses baseline config's
   traces sliced to the SAME test window as the tuned choice.

8. **Tie-break in best_params**: first in grid order (deterministic). Fallback
   when all grid points are below min_trades: default_post_params() with its
   score.

9. **Aggregation in report**: both mean-of-folds and trade-weighted reported.
