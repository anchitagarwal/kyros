## Findings

[HIGH] Repair #8 not landed â€” detectors re-run inside `_build_pools`
File: workspace/trading/snapshot.py:394 (also 424, 427); param declared :376, passed :283
Issue: `_build_pools` accepts a `precomputed` argument (swings/fvgs/order_blocks
  already computed in `build()` at lines 230/253/257) but the parameter is NEVER
  read in the method body. `_build_pools` re-runs `detect_swings(candles)` per TF
  at line 394 (duplicating line 230) and `detect_fvg`/`detect_order_blocks` on
  the HTF candles at lines 424/427 (duplicating the per-TF calls). So on every
  build, `detect_swings` executes twice per timeframe on identical candles, and
  the HTF FVG/OB detectors run an extra time. Invariant j #8 requires detectors
  "are not re-run on the same candles inside _build_pools / detect_fibonacci";
  `detect_fibonacci` honors the `swings=` passthrough, but `_build_pools` does
  not honor its own `precomputed` dict. A missed repair â†’ HIGH.
Fix: consume `precomputed["swings"]` (raw swings, not the serialized
  `_swing_dict` output â€” thread the raw `detect_swings` result, not
  `recent_swings`) for the per-TF swing pool sourcing, and thread the HTF
  `detect_fvg`/`detect_order_blocks` results (or the already-serialized
  `fvgs`/`order_blocks` reconstructed to the fields used) so no detector runs
  twice on the same candles. Add a call-count spy test (the blueprint promised
  one) to pin it. NOTE: this is behavior-neutral today (determinism/tests are
  unaffected because the detectors are pure), so it is a performance + invariant
  compliance defect, not a correctness regression.
