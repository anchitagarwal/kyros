## CONTINUATION INVARIANTS (i, j) — NOT IMPLEMENTED

The continuation spec states plainly: *"invariant i SUPERSEDES invariant e:
`_dol_target` is now expected to rank, not first-match; it should NOT match the
old nearest-only behavior."* The submitted code is exactly the old
first-match design. Independent grep across `workspace/trading/` and `tests/`:

```
grep -rn "rank_dols|score_pool|clarity_score|score_breakdown|ranked_dols" trading/  → 0 matches
grep -rln "clarity_score|ranked_dols|_rank_dols|_score_pool" tests/               → 0 matches
```

The blueprint itself promised `_DEFAULT_DOL_WEIGHTS`, `_DEFAULT_TF_WEIGHTS`,
`_DEFAULT_ROLE_WEIGHTS`, `ranked_dols_to_llm`, and `LiquidityPool.clarity_score
/ score_breakdown`; NONE are present in `config.py` or `snapshot.py`. The
executor's contract asserts "no implementation changes required" — that is
incorrect for this phase.

### [HIGH] Weighted clarity scoring absent — `_dol_target` still first-match
File: workspace/trading/snapshot.py:_dol_target / _nearest_internal / _nearest_external
Issue: `_dol_target` returns the first proximity-ordered match
(`_nearest_internal`/`_nearest_external` iterate `pools` pre-sorted by
`distance_points` and `return` the first hit). There is no `_score_pool`, no
`_rank_dols`, no `clarity_score`, no `score_breakdown`, no `ranked_dols`. This
is the entire point of the continuation phase — proximity DOES dominate today.
Input: a near weak 5m/low-confluence pool and a farther Weekly/Daily REL both
in the draw direction → the near pool wins. Invariant i requires the REL to win.
Fix: implement pure `_score_pool(pool, …) -> (score, breakdown)` summing
weighted tf/role/cycle_align/bias_align/confluence/pd_align/clean_path/proximity
(proximity LOW weight), a direction-filtered `_rank_dols` sorted by
`clarity_score` desc, set `dol_target == ranked_dols[0]` when cycle active,
expose `ranked_dols` on the snapshot + compact dict (top-N via
`ranked_dols_to_llm`), and add `clarity_score`/`score_breakdown` to
`LiquidityPool.to_dict`.

### [HIGH] `ranked_dols` field / serializer surface missing
File: workspace/trading/snapshot.py:MarketSnapshot / _compact_dict
Issue: invariant f (continuation) and i require `ranked_dols` in both `build()`
and `to_compact_dict()`, with `dol_target == ranked_dols[0]`. Neither exists.
LLM cannot see the ranked draw list.
Fix: add `ranked_dols: list[LiquidityPool]` to `MarketSnapshot`, populate in
`build()`, serialize top-N in `_compact_dict`.

### [HIGH] Proximity-does-not-dominate test absent (core acceptance test)
File: tests/phase2/test_fib_liquidity_cycle.py
Issue: no test asserts a farther Weekly/Daily REL beats a nearer weak pool, no
monotonicity tests, no `clean_path` opposing-ERL penalty test, no
`score_breakdown` sums-to-score test, no wrong-side-exclusion test for
`_rank_dols`. The phase's central guarantee is unverified.
Fix: add the weighted-DOL test battery per the blueprint's test matrix.

### [HIGH] Repair #1 not landed — fallback not ERL-only
File: workspace/trading/snapshot.py:_nearest_dol (used as fallback in _dol_target)
Issue: `_nearest_dol` filters only by `type`/`level`, not `scope`. Because IRL
pools are now tagged bsl/ssl via `_pool_type_for`, an INTERNAL pool can win the
fallback path (`_dol_target` → `_nearest_dol` when cycle None / no cycle match).
Repair #1 requires the fallback to run over the ERL-only set. `_nearest_dol`
itself stays byte-identical (correct), so the ERL-only filter must be applied to
the pool set passed in, not inside `_nearest_dol`.
Fix: pass an ERL-only (`scope == "external"`) pool list to the fallback in
`_dol_target`, leaving `_nearest_dol` byte-identical.

### [HIGH] Repair #7 not landed — prompt reads cycle AFTER direction
File: workspace/trading/reasoning_agent.py:ICT_SYSTEM_PROMPT (DOL-FIRST section)
Issue: the step ordering is literally `1.`, `2.`, then `1.5.`, then `3.` — the
"1, 2, 1.5, 3" anti-pattern repair #7 explicitly forbids. `liquidity_cycle` is
read at step 1.5 AFTER direction is chosen from `htf_bias` at step 2.
Fix: reorder so the cycle is read BEFORE direction selection; renumber
sequentially (no fractional step).

### [MEDIUM] Repair #5 dedup keeps distinct roles instead of merging richest
File: workspace/trading/snapshot.py:_build_pools (dedup by (level,type,role))
Issue: coincident ERL sources at the same `(level,type)` but different roles are
NOT merged; each survives as a separate pool. Repair #5 wants a single merged
pool keeping the richest role (equal>prior>session>swing) so confluence is not
double-counted. Under weighted scoring this inflates the candidate list and
double-counts confluence. (Low impact today only because scoring is absent.)
Fix: dedup on `(level,type)`, keep richest role, recount confluence on the
merged set.
