### (b) OutcomeSimulator lookahead safety — PASS (most critical) 
- **Step 1:** `simulate()` does NOT filter `subsequent_candles` internally by
  timestamp; the docstring states this is the caller's responsibility per spec.
  Consistent with spec; not a bug.
- **Step 2:** `BacktestEngine._slice_subsequent` (engine.py:~240) assembles
  subsequent candles with strict `if c_dt > alert_dt:`. The alert candle is in
  the replay buffer but excluded by the strict greater-than. The exact
  assembly line was located and verified.
- **Step 3 (my probe):** With `subsequent_candles=[T50,T51,T52,T53]` (alert
  candle T50 deliberately included), result = **win** (fill@T50, resolve@T51
  hitting target). With the correct `[T51,T52,T53]`, result = **expired**
  (fill@T51, resolution begins T52). The two differ — confirming that
  *including* the alert candle leaks lookahead. The engine does NOT include it,
  so the bug is absent. Documented both results.
- **Step 4 (my probe):** A post-fill candle straddling both target and stop
  returns **loss** with `actual_rr = -1.0`. Conservative resolution correct.
- **Step 5 (my probe):** `alert.bias == "no_trade"` returns
  `TradeOutcome(result="no_trade")` immediately, no candle iteration, all
  numeric fields None. Correct short-circuit.
- **Engine integration probe:** Instrumented the simulator to record every
  received candle timestamp across a full `BacktestEngine.run`; no alert
  timestamp appeared in the received set. No leak.

## Findings

### [MEDIUM] M1 — Disclaimer omits the LLM-training-data contamination caveat
File: `workspace/backtesting/report.py:38` (`_DISCLAIMER`)
Issue: Invariant (g) requires the report to disclose that results may be
optimistically biased because the LLM may have encountered this historical
period in its training data. The current disclaimer covers entry-mid fill
optimism and unmodeled slippage/commissions, but never mentions training-data
contamination — the single most important caveat for an LLM-driven backtest on
historical data the model likely saw. A reader could over-trust the win rate.
Fix: Add a sentence to `_DISCLAIMER` containing the contamination point, e.g.
"Results may be optimistically biased: the LLM may have seen this period in its
training data, so historical pattern recall cannot be distinguished from
genuine edge." (Including the phrase the spec greps for, e.g. "training data" /
"optimistically biased".)

### [MEDIUM] M2 — `backtest_report.md` not produced; grep target absent
File: (artifact) `workspace/backtest_report.md`
Issue: Invariant (g)'s file-level grep targets `workspace/backtest_report.md`,
which does not exist because no integration `generate()` run has been committed
this phase. The disclaimer/hash logic is unit-tested and correct, so this is
not a code defect, but the deliverable artifact is missing for an auditor doing
a file grep. (Severity MEDIUM, contingent on M1 — once M1 is fixed and a sample
report is emitted, both clear.)

### [LOW] L1 — Alpaca env vars not declared in `.env.example`
File: `.env.example`
Issue: `data_loader._load_alpaca` reads `ALPACA_API_KEY` / `ALPACA_SECRET_KEY`
from env, but `.env.example` declares neither (it documents
`KYROS_DATA_BACKEND` and `KYROS_CSV_PATH` only). The blueprint/contract called
for adding the empty-valued Alpaca keys. No secret is hardcoded (good), but the
self-documenting `.env.example` is incomplete for the alpaca path.
Fix: Add `ALPACA_API_KEY=` and `ALPACA_SECRET_KEY=` (empty) to `.env.example`.

### [LOW] L2 — Cache path drops backend from filename (collision risk)
File: `workspace/backtesting/data_loader.py:_cache_path`
Issue: Cache key is `NQ_1m_{start}_{end}.parquet` with no backend segment. The
blueprint explicitly warned that csv-derived and yfinance-derived caches must
not collide and suggested a backend suffix. The contract (D-spec) chose the
backend-less path "per spec", so this is an accepted deviation, but a csv cache
and a future yfinance cache for the same date range would alias to the same
file. Only csv is active this phase, so impact is currently nil.
Fix (optional): include `{backend}` in the cache filename to future-proof.
